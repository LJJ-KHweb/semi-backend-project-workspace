#  EVRE ERD 설계 의도

> 단순히 ERD를 나열하는 것이 아니라, **왜 이 테이블을 만들었고 왜 이 컬럼이 필요한지**까지 정리한 문서입니다.<br/>
> 사용자(`SEMI_USER`)를 중심에 두고, 모든 데이터가 "누가 만들었나"로 연결되는 **방사형 구조**로 설계했습니다.

##  설계를 관통하는 4가지 원칙

| 원칙 | 한 줄 요약 |
|:---|:---|
| **책임 분리** | 성격·변경 시점·권한이 다른 데이터는 테이블로 분리 |
| **Soft Delete** | 물리 삭제 대신 `STATUS='N'` → 복구 가능, 이력 보존 |
| **시퀀스 PK** | 동시 등록에도 중복 없는 고유 식별값 확보 |
| **1:N 정규화** | 파일·재고·충전기 등 여러 개 붙는 데이터는 분리 |

##  도메인별 설계 의도

<br/>

###  회원 · 인증 — `SEMI_USER` `SEMI_TOKEN`

> 서비스의 중심 테이블. 모든 활동 데이터가 여기로 연결됩니다.

**테이블을 나눈 이유**

회원 정보와 인증 정보를 한 테이블에서 관리하면, 토큰 재발급·만료 처리 같은 인증 기능이 회원 정보와 강하게 결합됩니다. 이를 방지하기 위해 **회원 정보와 인증 정보를 분리**하여 각각의 책임을 독립적으로 관리하도록 설계했습니다.

**컬럼을 둔 이유**

| 컬럼 | 이유 |
|:---|:---|
| `ROLE` | 사용자와 관리자의 권한을 구분 |
| `STATUS` | 회원 탈퇴 시 삭제 없이 상태만 변경하는 Soft Delete |
| `CREATE_DATE` | 가입 시점과 회원 활동 이력 확인 |
| `SEMI_TOKEN` (Refresh Token + 만료시간) | 로그인 상태를 안전하게 유지 |

<br/>

###  게시 기능 — `BOARD` `NOTICE` `REQUIRE`

> 셋 다 "게시글" 형태지만, 목적과 권한이 달라 분리했습니다.

**테이블을 나눈 이유**

일반 게시글·공지사항·문의사항을 하나의 테이블에서 관리하면 **권한과 비즈니스 로직이 복잡**해집니다. 목적과 권한이 서로 다르므로 세 테이블로 분리했습니다.

**컬럼을 둔 이유**

| 컬럼 | 이유 |
|:---|:---|
| `VIEWS` | 별도 로그 테이블 없이 조회수 관리 |
| `STATUS` | 삭제 데이터를 보존하는 Soft Delete |
| `PUBLIC_YN` | 공지사항만 공개 여부를 관리 (그래서 별도 설계) |
| 시퀀스 PK | 여러 사용자가 동시에 등록해도 번호 중복 방지 |

문의사항은 답변 여부와 연결되므로, 관리자 답변 기능을 고려해 설계했습니다.

<br/>

###  첨부파일 — `BOARD_FILE` `NOTICE_FILE` `REQUIRE_FILE`

> 게시글 하나에 파일 여러 개(1:N). 본문과 분리해 자유롭게 등록.

**테이블을 나눈 이유**

첨부파일을 게시글 테이블에 함께 저장하면 **파일 개수 제한이 생기고 구조 변경이 어려워집니다.** 그래서 파일을 별도 테이블로 분리해 하나의 게시글에 여러 파일을 자유롭게 등록하도록 했습니다.

또한 게시판·공지·문의는 수정/삭제 정책이 각각 다르므로, 공통 파일 테이블이 아닌 **기능별 파일 테이블**로 분리했습니다.

**컬럼을 둔 이유**

| 컬럼 | 이유 |
|:---|:---|
| `FILE_ORDER` | 업로드 순서 유지, 수정 시 순서 변경 가능 |
| `ORIGINAL_NAME` | 사용자가 올린 원본 파일명 그대로 표시 |
| `FILE_PATH` | 실제 파일은 서버에서 관리, DB 용량 증가 최소화 |

<br/>

###  문의 답변 — `ANSWER`

> 문의와 답변은 작성 시점·주체가 달라 분리.

**테이블을 나눈 이유**

문의와 답변은 **생성 시점과 작성 주체가 다르므로** 한 테이블에서 관리하지 않았습니다. 문의 등록과 답변 등록을 독립적으로 처리하고, 추후 여러 답변·답변 이력 기능으로 확장하기 쉽도록 설계했습니다.

**컬럼을 둔 이유**

- 답변 작성자와 작성 시점을 별도 관리 → **관리자 활동 이력** 확인
- 문의 번호(`REQUIRE_NO`)를 참조 → 문의와 답변을 연결

<br/>

###  상품 · 리워드 — `PRODUCT` `INVENTORY` `HISTORY_MILEAGE`

> 변경 시점과 관리 목적이 다른 셋을 분리.

**테이블을 나눈 이유**

상품 정보·재고·마일리지 내역은 **변경되는 시점과 관리 목적이 다릅니다.** 각각의 테이블로 분리해, 재고 변경이나 마일리지 적립이 상품 정보에 영향을 주지 않도록 설계했습니다.

**컬럼을 둔 이유**

| 항목 | 이유 |
|:---|:---|
| 재고(`INVENTORY`) 분리 | 상품 정보 수정 없이 수량만 변경 |
| 마일리지 내역(`HISTORY_MILEAGE`) | 누적 기록을 남기기 위해 별도 관리 |
| 상품 번호 시퀀스 | 삭제 여부와 무관하게 고유 식별값 유지 |

<br/>

###  차량 · 주행 — `CAR` `CAR_HISTORY` `RASP`

> 예약 정보 · 실시간 센서 · 누적 이력, 성격이 달라 3분할.

**테이블을 나눈 이유**

차량 예약 정보, 실시간 센서 데이터, 누적 이용 이력은 성격이 다르므로 각각 분리했습니다. **실시간 데이터와 서비스 데이터를 분리**해 조회 성능을 높이고 관리 책임을 명확히 했습니다.

**컬럼을 둔 이유**

| 테이블 | 이유 |
|:---|:---|
| `CAR` (시작·종료 시간) | 차량 예약 시간 중복 여부를 쉽게 검증 |
| `RASP` | 라즈베리파이 원본 데이터를 그대로 저장 |
| `CAR_HISTORY` | 누적 주행거리·전력량 저장 → 통계 시 실시간 데이터 반복 계산 방지 |

<br/>

###  충전소 · 충전기 — `STATION` `CHARGER`

> 충전소 하나에 충전기 여러 대(1:N).

**테이블을 나눈 이유**

하나의 충전소에 여러 충전기가 존재하므로 분리했습니다. 이를 통해 **충전기 추가·상태 변경이 충전소 정보에 영향을 주지 않도록** 설계했습니다.

**컬럼을 둔 이유**

| 컬럼 | 이유 |
|:---|:---|
| `LAT` / `LNG` | 지도 API에서 바로 사용 |
| CHARGER `STATUS` | 충전 가능 여부를 충전기별로 독립 변경 |
| 테이블 분리 | 충전기 수가 늘어도 충전소 테이블 구조 변경 불필요 |

---

##  관계 설계 요약

| 관계 | 형태 | 이유 |
|:---|:---:|:---|
| SEMI_USER ↔ 모든 활동 테이블 | 1:N | 데이터를 "누가 만들었나"로 추적 |
| STATION ↔ CHARGER | 1:N | 충전소 하나에 충전기 여러 대 |
| PRODUCT ↔ INVENTORY | 1:N | 상품 정보와 재고 분리 |
| 본문 ↔ 첨부파일 | 1:N | 게시글 하나에 파일 여러 개 |
| REQUIRE ↔ ANSWER | 1:N | 문의에 대한 관리자 답변 |
| SEMI_USER ↔ SEMI_TOKEN | 1:1 | 회원별 Refresh Token 관리 |
